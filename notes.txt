class - 5
==========

    -> Users - Groups - Roles(Permissions)


    -> User 

        -> Every Person who logs into servicenow should have a user created 

        -> Each user is a record in sys_user table(just like each incident in the incident table)

        -> A User record stores information such as

            -> User ID / Username (used for login)
            -> First Name 
            -> Last Name 
            -> Email 
            -> Phone No 
            -> Department
            -> etc 

        -> A user record also tracks 

            -> Which Groups the user belongs to 

            -> Which Roles the user has 

        -> NOTE: User -> belongs to Groups -> Inherits Roles (Recommended Way)

        -> NOTE: User -> Directly Assign Roles 


    -> Groups 

        -> For Organizing People 

        -> A Group is Collection Of Users 

        -> Groups records are stored in sys_user_group table 

        -> Groups act like buckets or teams that organize users who share a purpose 

            -> Service Desk 
            -> Network
            -> IT Security 
            -> etc 

    -> Why Groups are important 

        -> Assigning Work 

            -> Incidents, Problems, HR Cases etc can be Assigned to Groups 

            -> Any member of that group can pick up and start working on item 

        -> Managing Permissions 

            -> BEST PRACTICE is to assign Roles To Groups, not Directly to users 

            -> When a user is added to group, they will automatically inherit groups roles 

            -> This approach makes Onboarding, Transfers & Offboarding much easier.

        -> Notifications 

            -> You can send Notifications to an entire group, 
            Say Service Desk gets notified when P1 incident is logged

        -> Direct Role Assignment 

            ->  You go to a user record and add a role straight to their Roles tab. 
            It works, but it creates a management problem at scale.

            -> If you have 20 Service Desk agents and they all need the itil role, 
            you make 20 individual assignments. 
            When a new agent joins, you remember to add it manually. 
            When someone leaves or changes teams, you have to hunt down their user record and clean it up. 
            When an audit asks "who has itil access?", you need to scan every single user record.
        
        -> Via Group Assignment

            -> You add the itil role to the Service Desk group once, 
            then manage membership — not roles. 
            Every member of the group automatically inherits itil. 
            Add someone to the group, they instantly get the role. 
            Remove them from the group, the role is gone. The role itself never needs to be touched again.

    
    -> Roles - Granting Permissions 

        -> A Role is bundle of Permissions 

        -> Roles are records in the table sys_user_role 

        -> Think of Role as a key that will unlock certain actions to be done on servicenow 

        -> Roles can allow user to do 

            -> Read / Create / Update / Delete (CRUD)

        
        -> How Roles Can be Assigned 

            -> Role To User (Directly)

            -> Role To Group (All Group members inherit)

            -> Role To Role (Child Roles included automatically)


        -> NOTE: Assign Roles To Groups 
        -> NOTE: Avoid Assigning many roles Directly to user, as it becomes hard to manage and audit 

        -> OOB Roles(Out Of Box) 

            -> admin: Full Access to system 
            -> itil: Standard Role For IT Staffing manage incidents/problems/changes 
            -> knowledge_admin: Manages Knowledge Bases and Articles 
            -> catalog_admin: Manages Service Catalogs 
            -> report_admin: MAnage Reporting 
            -> cmdb_admin: Manages CMDB Configurations(ITSM & ITOM)

        
    -> NOTE: Putting All the things together

        -> User -> Group -> Role(Permission)

    -> NOTE: When we create a user, by default no Permissions are set for user

    -> IT Service Management (Zurich / Australia Version)


        -> Challenges Facing IT Organizations 

            -> When Considering IT Departments, the Four Primary Challenges 

        -> IT Service Management - Deliver IT Services 

            -> The Lifecycle of IT Services starts with Strategy and continues through 
            Design, Transition and Operation.

            -> The Key Phases in Delivering IT Services are i.e ITSM Life Cycle 

                -> PLAN -> DESIGN -> CREATE -> OPERATE -> IMPROVE 

            -> Supporting Applications Include 

                -> CMDB (Configuration Management Database)        

                -> Incident Management

                -> Problem Management 

                -> Change Management 

                -> Service Catalog 


            -> CMDB (Configuration Management Database) 

                -> Stores IT Assets 

            -> Incident Management

                -> Fix Issues 

            -> Problem Management

                -> Root Cause Issues 

            -> Change Management

                -> Control Changes 

            -> Service Catalog 

                -> User Requests 


    -> NOTE: Any Successful ITSM Programs has following practices i.e 4 P's 

        -> People -> Process -> Products -> Partners 

        -> People 

            -> Organization Structure to support Service Delivery 
            -> Training and Giving Skills Required to execute ITSM Process and Technology 
            -> Dedicated resources responsible for Managing Tasks 

        -> Process 

            -> Well defined ITSM Processes with clear definitions 
            -> Clear and Controlled Dependencies with other processes such as Asset Manage(ITAM)
            -> Process Owners and Stakeholders identified with clear responsibilities 

        -> Products

            -> Softwares that bring together the ITSM Lifecycle to improve the efficiency 
            -> Automation Applications for Discovering and Monitoring IT Services 
            -> Integrations with Applications used to support defined processes 

        -> Partners 

            -> Support the Organizations Business Model, providing 24 / 7 Support 
            -> Provide Technology that complements and enhances Organizations in-house Technology


    -> What are Benefits of ITSM 

        -> Benefits For Business 

            -> Increased Agility 

            -> Reduced Costs 

            -> Easy Compliance 

            -> Better Service 

        -> Benefits For IT 

            -> Improved Productivity 
            
            -> Increased User Satisfaction 

            -> Faster Response i.e Faster Incident Detection 

        -> Benefits For Employees 

            -> Improved IT Support 

            -> Clear Roles & responsibilities
            
            -> Improved Business Alignment 


    -> Terminology 

        -> Service 

            -> Delivers value by meeting a need or fulfilling a demand 

                -> Restaurant Providing Dining Services 

                    -> Table Setup 
                    -> Seating 
                    -> Wait Staff Greeting 
                    -> Order Taking 
                    -> Order Fulfillment 
                    -> Order Payment 
                    -> etc 

                    -> There are underlying processes

                        -> Hiring 
                        -> Training 
                        -> Technology 

        
        -> Request

            -> Mechanism for Obtaining, Changing or Discontinuing the use od Service 

                -> Request To Order Food 

                -> Request To Change the way cooking is done 

                -> Request To Discontinuing i.e didn't like the taste of food 

        -> Incident 

            -> An Unplanned Interruption or Quality Reduction 

                -> A Companies Financial System Stopped Responding 

        -> Problem 

            -> The underlying cause of one or more related incidents 

                -> The Company Financial System Stops Responding two or three time a week 

                -> Service is restored each time by rebooting the server on which this 
                Financial application resides 

                -> A Problem record is opened for multiple occurrences

                -> A Problem needs Detailed Investigation, identifying the "Root Cause" 
                which in our case is a Faulty Network Card in the Server,

                -> The Card is replaced and the incidents stop occurring 

            
        -> Change 

            -> The Addition, Modification or Removal of a service components 

                -> A company wide switch to a new instant messaging platform 
                i.e changing from Microsoft Teams To Slack 

                -> A Change to the server on which the instant messaging application resides on 

                    -> Big Infrastructure for Microsoft Teams 

                    -> Now Change To Small Infrastructure for Slack Application 


    -> NOTE: ITSM platform is structured into Three Tiers 

        -> ITSM Standard 

            -> Incident Management

            -> Change Management

            -> Asset Management 

            -> Request Management 

            -> Cost Management 

            -> Agent Workspace (Basic Gen AI)

        -> ITSM Professional (adds to Standard) (Intelligent Automation)

            -> Virtual Agent with AI 

            -> Predictive Intelligence (automatically directs tasks to most suitable team)

            -> AI Based Recommendations 

            -> Vendor Management Workspace 

        -> ITSM Enterprise (adds to Pro + Standard) (Optimization)

            -> Process Mining 

                -> Enables Visual Analysis of Process FLows to Identify Bottlenecks 
                and Inefficiencies to streamline all Business workflows 

            -> Workforce Optimization 

                -> Allocates Tasks Based On Agent Skills, Availability and Workload
